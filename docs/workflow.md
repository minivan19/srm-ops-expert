# SRM运维报告生成 - 完整流程文档

## 一、整体架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                        数据层                                        │
│  原始Excel工单 → extract_module_data.py → 完整工单数据             │
└─────────────────────────────────────────────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────────────┐
│                        LLM层                                         │
│  generate_report_v2.py → 4批次调用 → 结构化输出                    │
└─────────────────────────────────────────────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────────────┐
│                        输出层                                        │
│  md2docx.py → Word文档 → 发送用户                                  │
└─────────────────────────────────────────────────────────────────────┘
```

---

## 二、数据层流程

### 2.1 数据来源
- **路径**: `C:\Users\mingh\client-data\raw\客户档案\{客户}\运维工单\`
- **格式**: Excel文件（多个文件合并）

### 2.2 关键字段（完整7个字段）
| 字段 | 说明 | 给LLM |
|------|------|--------|
| 工单号 | 编号 | ✅ |
| 分类 | 问题类型 | ✅ |
| 模块 | 系统模块 | ✅ |
| 标题 | 工单摘要 | ✅ |
| 描述 | 详细描述 | ✅ |
| 解决方案 | 处理方案 | ✅ |
| 根本原因 | 问题根因 | ✅ |

### 2.3 问题数量规则（动态）
| 工单数量 | 输出问题数 |
|---------|-----------|
| <10 | 不输出 |
| 10-19 | 1个 |
| 20-29 | 2个 |
| ≥30 | 3个 |

## 三、LLM层说明

### 3.1 4批次调用

| 批次 | 内容 | 数据来源 | 问题数量规则 |
|------|------|---------|-------------|
| 第0批 | 趋势+SLA分析 | 统计数据 | - |
| 第1批 | 分类+模块分析 | 完整工单数据 | 动态计算 |
| 第2批 | FAQ | 完整工单数据 | - |
| 第3批 | 总结 | 前两批结果 | - |

### 3.2 提示词约束

#### 第1批：分类+模块分析
```
### 重要约束
1. 只基于提供的工单数据进行分析，不要推测
2. 只分析工单数≥10的分类或模块
3. 问题数量规则：
   - 工单数10-19：分析1个问题
   - 工单数20-29：分析2个问题
   - 工单数≥30：分析3个问题
4. 解决方案必须基于工单中已有的"解决方案"字段，如果为空则写"待完善"
5. 禁止使用"优化"、"完善"、"加强"等空洞词汇
6. 参考工单必须与问题描述直接相关，禁止幻觉
```

#### 第2批：FAQ
```
### 重要约束
1. 基于工单的"标题"、"描述"、"解决方案"、"根本原因"四个字段综合分析
2. 禁止使用"优化"、"完善"等空洞词汇
3. 解决方案必须具体可执行，如果为空写"待完善"或"联系管理员"
4. 问题场景需要聚类提炼，不强求使用原文
5. 参考工单必须与问题描述直接相关，禁止幻觉
```
