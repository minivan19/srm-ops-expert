- ✅ 所有分析步骤都有明确的模板和规则

## 第二阶段：AI 分析执行（确定性流程）

### 2.1 第一部分：工单概览

#### 2.1.1 1.4 趋势分析（确定性模板）

**输入数据**：月度工单趋势表

**分析步骤**（必须按此顺序执行）：

**步骤 1：整体趋势判断**
- 找出工单量最高的 3 个月
- 找出工单量最低的 3 个月
- 计算波动幅度：(最高月工单量 - 最低月工单量) / 最低月工单量

**步骤 2：阶段划分（固定规则）**
- 高峰期：工单量 > 全年平均值 × 1.2
- 低谷期：工单量 < 全年平均值 × 0.8
- 平稳期：其他月份

**步骤 3：原因分析（标准化推理）**
- 高峰期原因（按优先级选择）：
  1. 季度末/半年报/年底结算
  2. 系统重大版本更新
  3. 新功能上线
  4. 业务旺季
- 低谷期原因（按优先级选择）：
  1. 春节/国庆等长假
  2. 业务淡季
  3. 系统稳定期

**输出格式（必须严格遵循）**：

```
【1.4 趋势分析】

**整体趋势判断**
本年度工单量呈现[描述：持续上升/先升后降/波动较大/相对平稳]特征。
- 最高峰：[月份]，[数量]单
- 最低谷：[月份]，[数量]单
- 波动幅度：[百分比]

**阶段划分**
根据数据分析，自然分为以下阶段：

1. **高峰期**：[月份列表]
   - 特征：工单量显著高于平均水平
   - 可能原因：[从标准化原因中选择 2-3 个]

2. **低谷期**：[月份列表]
   - 特征：工单量显著低于平均水平
   - 可能原因：[从标准化原因中选择 1-2 个]

3. **平稳期**：[月份列表]
   - 特征：工单量在平均水平附近波动

**趋势预测与建议**
基于历史数据，建议：
1. 在高峰期前[1个月]增加运维资源准备
2. 在低谷期开展系统优化和用户培训
3. 针对[高峰期原因]制定专项应对方案
```

#### 2.1.2 1.5 SLA 与满意度分析（确定性模板）

**输入数据**：SLA 达标率、产品满意度、服务满意度

**分析步骤**（必须按此顺序执行）：

**步骤 1：数据解读**
- SLA 达标率：≥98% 为优秀，95-98% 为良好，<95% 需要改进
- 满意度：≥4.5 为优秀，4.0-4.5 为良好，<4.0 需要改进

**步骤 2：关联分析**
- 如果 SLA 达标率高但满意度低：分析响应速度 vs 解决质量
- 如果 SLA 达标率低但满意度高：分析用户期望管理
- 如果两者都低：识别系统性问题

**输出格式（必须严格遵循）**：

```
【1.5 SLA 与满意度分析】

**核心指标表现**
- SLA 达标率：[数值]% → [评价：优秀/良好/需改进]
- 产品满意度：[数值]分 → [评价：优秀/良好/需改进]
- 服务满意度：[数值]分 → [评价：优秀/良好/需改进]

**综合分析**
[根据关联分析结果，选择以下一种]

**场景 A：两者均优秀**
本年度 SLA 达标率和满意度均表现优秀，说明：
1. 运维响应及时且问题解决质量高
2. 用户对产品和服务整体满意度高
3. 运维流程和服务标准执行到位

**场景 B：SLA 优秀，满意度待提升**
SLA 达标率优秀但满意度有待提升，可能原因：
1. 响应及时但问题解决不够彻底
2. 沟通反馈机制需要优化
3. 用户期望管理需要加强

**场景 C：SLA 待提升，满意度优秀**
SLA 达标率有待提升但满意度优秀，说明：
1. 用户对服务过程满意，但响应时效有改进空间
2. 服务态度和沟通弥补了时效的不足
3. 建议优化流程，在保持服务质量的同时提升响应速度

**场景 D：两者均待提升**
SLA 达标率和满意度均有待提升，需关注：
1. 识别影响响应时效的瓶颈环节
2. 分析问题解决质量的系统性问题
3. 建立服务质量监控和持续改进机制

**改进建议**
1. [针对主要问题，提出 1-2 个具体改进方向]
2. [针对次要问题，提出 1 个优化建议]
```

### 2.2 第二部分：分类维度深度分析

#### 2.2.1 核心问题筛选标准（确定性规则）

**必须严格遵循以下标准筛选问题**：

**优先级排序规则**：
1. 首先按工单数量降序排列（高频问题优先）
2. 相同数量时，按影响范围排序（多模块受影响优先）
3. 相同影响范围时，按严重程度排序（报错 > 操作问题 > 咨询）

**问题数量要求**：
- 每个分类必须生成 **3-5 个** 核心问题
- 工单量 ≥ 30 单的分类：3 个问题
- 工单量 20-29 单的分类：2 个问题
- 工单量 10-19 单的分类：1 个问题
- 工单量 < 10 单的分类：不输出

**问题质量要求**：
- 问题必须有具体的工单样本支持
- 问题必须有明确的原因分析
- 解决方案必须可执行、可验证

---

#### 2.2.2 问题分析格式（标准化模板）

**每个问题必须严格遵循以下格式**：

```
===问题分隔线===
问题：[简洁明了的问题标题，15-25字]
原因：[详细分析问题产生的原因，2-3句话，包含技术原因和管理原因]
解决方案：
1. [场景描述]：[操作路径/步骤，包含具体模块名称用【】括起来]，[注意事项]
2. [场景描述]：[操作路径/步骤，包含具体模块名称用【】括起来]，[注意事项]
3. [场景描述]：[操作路径/步骤，包含具体模块名称用【】括起来]，[注意事项]
参考工单：[相关的1-3个工单编号，用逗号分隔]
```

**解决方案必须包含**：
- 具体场景描述
- 操作路径/步骤（包含模块名称）
- 注意事项

**禁用词汇**（必须替换为具体描述）：
- ❌ "提供" → ✅ "在【模块名】中配置"
- ❌ "优化" → ✅ "调整【模块名】中的参数"
- ❌ "加强" → ✅ "增加【模块名】的检查"
- ❌ "完善" → ✅ "补充【模块名】中的内容"

### 2.4 第四部分：高频问题 Q&A 知识库

#### 2.4.1 FAQ 筛选标准（确定性规则）

**必须严格遵循以下标准**：

**筛选范围**：
- 仅从"业务咨询"分类的工单中筛选
- 排除报错类、故障类工单
- 重点选择：查询类、操作类、流程类问题

**优先级排序**：
1. 首先按问题重复度降序排列（多人问过的优先）
2. 相同重复度时，按普适性排序（适用于多个模块的优先）
3. 相同普适性时，按实用性排序（日常高频使用的优先）

**数量要求**：
- 必须生成 **5 个** FAQ（不能多也不能少）

---

#### 2.4.2 FAQ 格式（标准化模板）

**每个 FAQ 必须严格遵循以下格式**：

```
===问题分隔线===
问题：[用户常见问题，以"如何"、"怎么"、"什么是"开头]
适用场景：[说明此问题在什么情况下会遇到，1-2句话]
解决方案：[详细的操作步骤，包含：1. 2. 3.，每个步骤包含具体模块名称【】]
参考工单：[相关的1-2个工单编号，用逗号分隔]
```

### 3.3 验证失败处理流程

**如果验证检查失败，必须按以下流程处理**：

1. **识别缺失内容**：定位哪些占位符未替换
2. **重新执行分析**：针对缺失部分重新执行 AI 分析
3. **再次验证**：完成补充后重新运行验证检查
4. **最多重试 2 次**：如果 2 次重试仍失败，记录问题并继续
